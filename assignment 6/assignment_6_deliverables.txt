Assignment 6 Deliverables: Unified Customer Support Agentic System
=================================================================

Objective
---------
Build a unified customer support system that combines multiple agentic design patterns: Tool Use, Planning, Reflection, and Multi-Agent collaboration. The system retrieves customer issue data, creates a resolution plan, refines the plan, and returns a support-ready final output.

1. Python Script
----------------
The implementation is contained in the following files:

- src/customer_support_system/agents.py
- src/customer_support_system/main.py

agents.py defines the Tool Use Agent, Planning Agent, Reflection Agent, and CustomerSupportOrchestrator. main.py provides the command-line interface for running one or more customer support cases.

Tool Use Agent:

```python
class ToolUseAgent:
    def fetch_customer_data(self, customer_id: str):
        print(f"Tool Use Agent: fetching customer data for {customer_id}...")
        data = CUSTOMER_DATABASE.get(customer_id)

        if not data:
            return {"error": "Customer not found."}

        return CustomerRecord(customer_id=customer_id, **data)
```

Planning Agent:

```python
class PlanningAgent:
    def __init__(self, llm):
        self.llm = llm

    def create_resolution_plan(self, customer_record):
        prompt = planning_prompt_template.format_prompt(
            customer_data=customer_record.to_prompt_text(),
        ).to_messages()
        response = self.llm.invoke(prompt)
        return response.content.strip()
```

Reflection Agent:

```python
class ReflectionAgent:
    def __init__(self, llm):
        self.llm = llm

    def refine_resolution_plan(self, customer_record, initial_plan):
        prompt = reflection_prompt_template.format_prompt(
            customer_data=customer_record.to_prompt_text(),
            initial_plan=initial_plan,
        ).to_messages()
        response = self.llm.invoke(prompt)
        return response.content.strip()
```

Multi-agent orchestration:

```python
def run(self, customer_id: str) -> SupportWorkflowResult:
    customer_data = self.tool_use_agent.fetch_customer_data(customer_id)

    if isinstance(customer_data, dict) and "error" in customer_data:
        return SupportWorkflowResult(
            customer_record=None,
            initial_plan="",
            refined_plan="",
            error=customer_data["error"],
        )

    initial_plan = self.planning_agent.create_resolution_plan(customer_data)
    refined_plan = self.reflection_agent.refine_resolution_plan(
        customer_data,
        initial_plan,
    )

    return SupportWorkflowResult(
        customer_record=customer_data,
        initial_plan=initial_plan,
        refined_plan=refined_plan,
    )
```

4. Architecture and Workflow Explanation
----------------------------------------
This system uses a unified agentic workflow for customer support. The workflow begins with a customer ID and passes information through several specialized agents.

Workflow:

Customer ID
-> Tool Use Agent retrieves customer data from a simulated support database
-> Planning Agent creates an initial resolution plan
-> Reflection Agent reviews and improves the plan
-> Orchestrator returns the final support-ready output

The Tool Use Agent demonstrates the Tool Use Pattern by simulating an API or customer support database lookup. In a real system, this could be replaced with a Zendesk, Salesforce, HubSpot, Freshdesk, or internal CRM API call.

The Planning Agent demonstrates the Planning Pattern by turning the customer issue into a structured support workflow. It creates an issue summary, priority assessment, resolution steps, communication plan, escalation recommendation, and follow-up timeline.

The Reflection Agent demonstrates the Reflection Pattern by reviewing the initial plan and refining it for clarity, urgency, empathy, and operational usefulness. This improves the quality of the final support response.

The overall system demonstrates the Multi-Agent Pattern because the workflow is split into separate agents, each with a focused responsibility. The orchestrator coordinates these agents and passes each output into the next stage.

5. Reflection Report
--------------------
This assignment shows how multiple agentic design patterns can be combined into a single practical customer support workflow. Instead of asking a chatbot to answer a support issue directly, the system follows a structured process: it retrieves customer data, plans a resolution, reflects on the plan, and returns a refined final output.

This structure improves efficiency because the support agent receives a ready-made plan rather than having to manually review the issue from scratch. It also improves consistency because each case is processed through the same workflow. High-priority issues can be handled with escalation and faster follow-up timelines, while lower-priority issues can receive a more measured response.

The simulated API supports realistic tool use without requiring a live support platform. It allows the system to demonstrate how customer data can be retrieved and passed into the planning and reflection stages. If this system were connected to a real support platform, the simulated customer database could be replaced with an actual API client while keeping the rest of the workflow mostly unchanged.

The main challenge in this design was making sure each agent had a clear responsibility. If too much work is placed into one prompt, the system becomes harder to understand and debug. Separating the agents makes the workflow more modular and easier to extend.

6. Results Analysis
-------------------

Efficiency:
The system reduces response preparation time by automatically generating a structured resolution plan from customer data.

Scalability:
The system can process multiple customer IDs from a comma-separated command-line input. With a real API, the same design could scale to many tickets.

Adaptability:
The system handles different support scenarios by using the customer issue, priority, sentiment, and previous contact history. It also includes fallback handling when a customer ID is not found.

7. Reflection Questions
-----------------------

Question 1: How does combining Tool Use, Planning, Reflection, and Multi-Agent patterns improve the system?

Combining the patterns makes the system more reliable and useful than a single chatbot response. Tool Use provides customer data, Planning turns that data into a structured resolution workflow, Reflection improves the plan, and the Multi-Agent design keeps each responsibility separate. This creates a clearer, more maintainable, and more practical support system.

Question 2: How does the simulated API support the Tool Use pattern?

The simulated API acts like a customer support database. The Tool Use Agent receives a customer ID and retrieves the matching customer record. This mirrors how a real support system would fetch data from a CRM or ticketing platform. The rest of the workflow can then use that retrieved data to create a more relevant and personalized response.

Question 3: How could this system connect to a real support platform?

The simulated customer database could be replaced with a real API client for a platform such as Zendesk, Freshdesk, Salesforce, HubSpot, or an internal ticketing system. The Tool Use Agent would make authenticated API requests, retrieve live customer and ticket data, and pass the result into the same Planning and Reflection agents. Additional features could include ticket updates, automatic escalation, SLA tracking, and customer response drafting.
